# Test Runner CLI 功能说明

## 项目简介

Test Runner 是一个基于 Rust 编写的 HTTP 集成测试框架 CLI 工具。它允许开发者通过声明式的 YAML DSL 来编写、组织和执行针对 HTTP API 的集成测试，支持多阶段测试场景、Mock 服务器、数据库操作和 Redis 操作，无需编写任何代码即可完成复杂的测试流程。

---

## 功能概览

### 1. CLI 命令系统

- **`init`** — 初始化测试项目脚手架，在目标项目中生成 `.testrunner/` 目录结构及示例文件
- **`schema [KIND]`** — 生成 project / env / api / case / workflow / mock-route 的 JSON Schema，方便 AI Agent 或外部校验器做结构校验
- **`test api <API_ID>`** — 运行指定 API 的测试用例
- **`test dir <DIR>`** — 运行指定目录下的所有测试用例
- **`test all`** — 运行项目中的全部测试用例
- **`test workflow <WORKFLOW_ID>`** — 运行指定的工作流定义

### 2. 测试用例 DSL

提供基于 YAML 的领域特定语言，用于声明式地定义测试用例，支持以下步骤类型：

- **HTTP 请求（Request）** — 发送 HTTP 请求并对响应进行断言
- **SQL 执行（Sql）** — 对 MySQL / PostgreSQL 数据库执行 SQL 脚本
- **数据库查询（QueryDb）** — 查询数据库并提取 / 断言查询结果
- **Redis 操作（Redis / QueryRedis）** — 执行 Redis 命令并处理响应
- **变量设置（Set / UseData）** — 设置变量或从数据文件加载测试数据
- **条件分支（Conditional）** — 基于表达式执行 if / then / else 逻辑
- **循环迭代（Foreach）** — 遍历数组并对每个元素执行步骤

### 3. 断言系统

支持多种断言操作符来验证测试结果：

- 相等 / 不等（`eq`、`ne`）
- 包含（`contains`）
- 非空 / 存在（`not_empty`、`exists`）
- 数值比较（`gt`、`ge`、`lt`、`le`）

### 4. 表达式与模板引擎

- **模板渲染** — 在字符串中使用 `{{ expression }}` 语法进行变量替换
- **路径导航** — 支持嵌套数据访问，如 `response.json.items[0].name`
- **条件表达式** — 支持比较运算符（`==`、`!=`、`>`、`>=`、`<`、`<=`）
- **内置函数** — 如 `len()` 获取数组、对象或字符串的长度
- **字面量支持** — `true`、`false`、`null`、字符串、数字

### 5. 项目配置管理

- **项目配置（project.yaml）** — 定义项目元信息、默认值和 Mock 服务器设置
- **多环境配置** — 支持按环境（如 `local`、`ci`）配置不同的 Base URL、请求头和变量
- **API 定义** — 集中管理 API 的 HTTP 方法、路径、默认请求头和请求体
- **数据源目录** — 配置 MySQL、PostgreSQL、Redis 连接信息
- **测试数据** — 从 `data/` 目录加载 JSON / YAML 文件构建嵌套数据树

### 6. 内嵌 Mock 服务器

- 启动基于 Axum 的本地 HTTP Mock 服务器
- 根据请求方法和路径匹配 Mock 路由
- 支持从请求中提取值用于动态响应
- 支持条件响应（根据请求内容返回不同结果）
- 支持模板变量替换（在响应头和响应体中使用 `{{ vars.xxx }}`）
- 支持从文件加载响应体
- 支持多路由优先级排序
- 提供优雅关闭机制

### 7. 工作流编排

- **顺序执行** — 按定义顺序依次执行多个测试用例
- **数据传递** — 在工作流步骤之间传递导出数据
- **条件分支** — 根据前序步骤结果执行不同的用例
- **延迟清理** — 在工作流结束时统一执行 Teardown，而非每个用例单独清理
- **状态追踪** — 跟踪每个工作流步骤的状态、通过情况和导出数据

### 8. 测试执行引擎

- **用例选择与过滤** — 按 API、目录、标签（Tag）或用例 ID/名称筛选
- **Setup / Teardown** — 每个测试用例支持前置和后置步骤
- **Fail-Fast 模式** — 遇到首个失败即停止执行
- **Dry-Run 模式** — 仅展示执行计划而不实际运行
- **连接池管理** — 自动管理 MySQL、PostgreSQL、Redis 连接池
- **HTTP 客户端** — 支持超时、自定义请求头、请求体和查询参数

### 9. 测试报告

- **多种输出格式** — 支持 Summary（摘要）、JSON.
- **详细报告** — 记录每个步骤的类型、状态、耗时和错误信息
- **报告持久化** — 自动将结果写入 `.testrunner/reports/last-run.json`
- **退出码** — 有用例失败时返回非零退出码，便于 CI/CD 集成

### 10. 项目脚手架初始化

通过 `init` 命令一键生成完整的测试项目结构，包括：

- 环境配置模板（Local / CI / Minimal）
- 示例 API 定义和测试用例
- 数据源配置模板（MySQL、PostgreSQL、Redis）
- Mock 服务器路由和 Fixture 模板
- 工作流定义示例
- 测试数据目录和示例文件
- 测试报告输出目录

## 当前不足与待改进项

### 1. 副作用捕获依赖侵入式探针

当需要验证被测项目对 Redis、MySQL 等产生的副作用时，目前缺乏非侵入式的观测手段，需要在被测项目中安装切面探针。已调研的几种方案：

| 方案 | 说明 |
|------|------|
| **eBPF** | 内核级追踪，非侵入但实现复杂度高 |
| **Redis / MySQL Proxy** | 在中间层代理中拦截和记录操作 |
| **Redis / MySQL 日志收集** | 通过数据库自身的日志输出能力进行收集 |
| **代码中安装 SDK** | 针对不同框架注入程序输出日志；但对于 Swoft 等使用 PHPRedis 的项目，缺乏统一切入点，难以实现 |

**当前推荐方向**：利用 Redis / MySQL 自身的日志输出能力，配合日志收集来实现副作用观测，避免对业务代码的侵入。

### 2. AI 驱动的测试报告尚未实现

报告模块计划结合 AI 能力对测试结果进行智能分析和总结输出，但该功能目前尚未实现。
